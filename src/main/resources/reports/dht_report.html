Обзор проекта

Что такое DHT?

DHT (Distributed Hash Table) - это распределенная система, которая обеспечивает поиск данных в сети без центрального сервера. Используется в BitTorrent, Ethereum и других P2P системах.

Что это за проект? 🤔

Представьте, что у вас есть огромная библиотека книг, но книги разбросаны по тысячам домов в городе. Как найти нужную книгу? DHT (Distributed Hash Table) - это как умная система каталогов, которая помогает быстро найти любую "книгу" (данные) в этой распределенной "библиотеке" (сети компьютеров).

Ваш проект - это попытка создать такую систему на основе алгоритма Kademlia, который используется в BitTorrent, Ethereum и других крутых технологиях.

  • XOR метрика: Метод computeDistanceAsUnionId возвращает null, нет сравнения идентификаторов
  • K-buckets: Метод checkCapacity пустой - нет логики LRU, нет обновления узлов
  • Алгоритм поиска: Метод findClosestNode возвращает null, нет итеративного алгоритма
  • RPC операции: Отсутствуют PING, STORE, FIND_NODE, FIND_VALUE
  • Сетевое взаимодействие: Нет транспортного слоя, сериализации/десериализации

Теория Kademlia:

  • XOR метрика: Определяет расстояние между узлами как XOR их идентификаторов, интерпретируемый как целое число
  • K-buckets: Каждый узел хранит k-buckets для каждого бита в идентификаторе, содержащие до k узлов
  • Таблица маршрутизации: Организована как бинарное дерево с k-buckets в листьях
  • Алгоритм поиска: Итеративный поиск с параллельными запросами к α узлам (обычно α=3)
  • RPC операции: PING, STORE, FIND_NODE, FIND_VALUE - основа протокола

Архитектура проекта

Архитектурный подход

Проект использует Clean Architecture с четким разделением на слои:

  • Domain - бизнес-логика DHT
  • Application - приложение и сервисы

Ключевые компоненты

  • node - узлы сети
  • router - маршрутизация
  • storage - хранилище
  • protocol - протокол
  • messages - сообщения
  • id - идентификаторы

Типы узлов

  • KademliaRegularNode - обычный узел
  • KademliaRoutingNode - маршрутизирующий узел
  • KademliaStorageNode - узел хранения

Анализ алгоритмической части проекта DHT

Требования к реализации Kademlia (из оригинальной статьи)

Основные компоненты:

  1. 160-битные идентификаторы - ключи и узлы
  2. XOR метрика для вычисления расстояния
  3. k-buckets - списки узлов для каждого уровня дерева
  4. Четыре основных RPC операции:
  5. PING - проверка доступности узла
  6. STORE - сохранение пары (ключ, значение)
  7. FIND_NODE - поиск k ближайших узлов к заданному ID
  8. FIND_VALUE - поиск значения по ключу

Алгоритм поиска:

  1. Итеративный поиск с параллельными запросами к α узлам (обычно α=3)
  2. Из ответов выбираются k ближайших узлов к цели
  3. Процесс повторяется до нахождения k ближайших узлов

Таблица маршрутизации:

  • Массив из 160 k-buckets (по одному на каждый бит)
  • Каждый k-bucket содержит до k узлов (обычно k=20)
  • LRU политика замещения при переполнении

Сравнение с реализацией в проекте

✅ Что реализовано правильно:

  1. KademliaId: 160-битные идентификаторы (BIT_COUNT = 160)
  2. XOR метрика: Метод computeDistance корректно реализован
  3. Структура k-buckets: Класс Bucket и XorTreeRoutingTable
  4. Алгоритм поиска bucket: Метод getBucketNumberForUnid корректен

❌ Критические пробелы:

  1. Отсутствуют RPC операции: PING, STORE, FIND_NODE, FIND_VALUE не реализованы
  2. Нет итеративного поиска: Метод findClosestNode возвращает null
  3. Нет логики k-bucket: Метод checkCapacity пустой, нет LRU
  4. Нет сетевого взаимодействия: Отсутствует транспортный слой
  5. Нет алгоритма присоединения к сети: Отсутствует bootstrap процедура

🔧 Частично реализовано:

  1. Таблица маршрутизации: Структура есть, но логика неполная
  2. Система сообщений: Есть пакеты protocol/ping и protocol/find, но содержимое минимальное
  3. Хранилище: Интерфейсы есть, реализация отсутствует

Полный анализ пробелов в реализации DHT

1. Критические пробелы (блокируют функциональность)

1.1. XOR метрика и система идентификаторов

  • ❌ Метод computeDistanceAsUnionId в KademliaId возвращает null
  • ❌ Отсутствует сравнение идентификаторов по расстоянию (нет метода типа compareTo)
  • ❌ Нет методов для определения общего префикса идентификаторов

1.2. K-buckets и таблица маршрутизации

  • ❌ Метод checkCapacity в Bucket пустой - нет логики LRU

  • ❌ Нет логики разделения переполненных k-buckets

1.3. Алгоритм поиска

  • ❌ Метод findClosestNode в KademliaRegularNode возвращает null

  • ❌ Нет параллельных запросов к α узлам
  • ❌ Нет логики выбора k ближайших узлов

public UnionId computeDistanceAsUnionId(UnionId other) {
    byte[] thisBytes = this.getBytes();
    byte[] otherBytes = other.getBytes();
    byte[] result = new byte[thisBytes.length];

    for (int i = 0; i < thisBytes.length; i++) {
        result[i] = (byte) (thisBytes[i] ^ otherBytes[i]);
    }

    return new KademliaId(result);
}

private void checkCapacity() {
    if (nodeInfoSet.size() >= maxCapacity) {
        // Найти самый старый узел
        NodeInfo oldestNode = findOldestNode();

        // Пинговать его
        if (!pingNode(oldestNode)) {
            // Если не отвечает, удалить
            nodeInfoSet.remove(oldestNode);
        } else {
            // Если отвечает, отклонить новый узел
            throw new BucketFullException();
        }
    }

public Set<NodeInfo> findClosestNode(UnionId targetId) {
    Set<NodeInfo> closest = routingTable.findClosestNodes(targetId);
    Set<NodeInfo> queried = new HashSet<>();

    while (true) {
        // Выбрать α узлов для запроса
        Set<NodeInfo> toQuery = selectAlphaNodes(closest, queried);

        if (toQuery.isEmpty()) break;

        // Параллельные запросы
        CompletableFuture<Set<NodeInfo>>[] futures = 
            toQuery.stream()
                   .map(node -> findNodeAsync(node, targetId))
                   .toArray(CompletableFuture[]::new);

        // Обработать результаты
        for (CompletableFuture<Set<NodeInfo>> future : futures) {
            try {
                Set<NodeInfo> result = future.get(TIMEOUT, TimeUnit.SECONDS);
                closest.addAll(result);
            } catch (TimeoutException | ExecutionException e) {
                // Обработать ошибку
            }
        }

        queried.addAll(toQuery);
    }

    return selectKClosest(closest, targetId);
}
